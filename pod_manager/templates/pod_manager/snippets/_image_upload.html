{% comment %} THE image upload widget. Every image upload in the app renders through this: network fallback image, network logo, the 404 pool, network + user mix covers, and the custom avatar. There were four hand-rolled copies before this, and only one of them had a progress bar. Params: name file input's name (required) label field label current_url existing image, if any — rendered as a preview help_text small print under the field accept file accept list (default image/*) required mark the file input required (an add-form where the image IS the row; a replace-an-existing-image field must not be) input_class classes for the file input, when the host page styles its own (the user-mix modals use .modal-dark-input) show_remove render a "Remove" submit that posts remove=1 to the SAME action. Only for a form whose handler understands remove=1 — i.e. a form that is ONLY this image. Never in a form with other fields; it would submit them too. remove_confirm confirm() text for that button preview_px preview box size (default 96) processing_label the >=100% state (default "Processing image…") The progress bar itself is NOT here — it lives in snippets/_upload_progress.html, shared with the audio upload. This snippet is the image-specific chrome (preview, accept list, remove) wrapped around it; the data-upload-scope marker below is how that snippet finds this widget's own file input. IT IS A FIELD, NOT A FORM. Most consumers embed it in a bigger form (the mix modals, the 404 add row, the avatar picker), and a nested
is invalid HTML. It binds to its ENCLOSING form, which MUST be: and must NOT set hx-boost="false". Both attributes are load-bearing: - enctype the no-JS submit, and what makes it a real multipart POST; - hx-encoding makes htmx send the form as FormData over XHR. A multipart form that boosts WITHOUT it silently drops the file — there is a test for that. Boosting is what keeps the out-of-region audio player alive on save, and the XHR is also what makes the progress bar below possible at all: a browser shows NO progress for a native form POST. See docs/boosted-navigation.md. This is why the widget does NOT hand-roll its own XHR the way tab_notfound.html used to: doing so meant hx-boost="false", a BoostOptOutContractTests allowlist entry, and a hand-written htmx.ajax() region swap to avoid hard-navigating on success. Riding htmx's own XHR gets identical progress with none of that — and took both of the allowlist's entries back out. {% endcomment %}
